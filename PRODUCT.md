# Product

## Register

product

## Users

第一版用户是准备 Java 后端技术面试的候选人，尤其是需要把 Java 基础、JVM、并发、Spring、数据库、分布式和项目经历系统化复习的人。

他们通常在面试前数天到数周使用产品，需要快速发现薄弱点、主动复述答案、追踪历史表现，并按复习计划回到最容易失分的问题。

后续用户范围可以扩展到前端、全栈、Go、Python、算法、AI 工程、产品/行为面试等方向，因此产品语言和信息架构不能把 Java 写死为唯一身份。

## Product Purpose

面试雷达是一个 AI 辅助面试复习 Web/PWA 平台。它把题库、AI 生成、练习评分、复习调度、RAG 个性化上下文、学习分析和持久化数据组织成一个真实学习闭环。

成功状态不是“能和 AI 聊天”，而是用户能看到自己今天该复习什么、为什么该复习、哪些分类薄弱、每次回答哪里缺失，以及下一次什么时候回来复习。

## Brand Personality

清醒、专业、有一点锋芒。

界面应该像一个高效的面试作战工作台：信息明确、反馈及时、对候选人有压力感但不焦虑。首页和知识图谱可以更有表现力；练习页和 Dashboard 必须安静、可读、快速。

## Anti-references

不要做成普通 AI 聊天套壳、营销感过强的落地页、只展示大标题和装饰卡片的空壳产品，也不要做成花哨但影响复习效率的动效展示。

避免：

- 静态题库网站，只能看题和答案。
- ChatGPT 包装壳，缺少练习记录、复习调度和用户数据。
- 过度使用动画影响输入、表单、长列表和高频学习操作。
- 把 Java 写死进核心模型，导致后续无法扩展其他面试方向。
- 缓存或展示敏感简历、JD、回答内容时缺少明确的数据隔离和同步策略。

## Design Principles

1. 复习效率优先：用户每次打开都应该知道下一步练什么。
2. 数据驱动信任：用真实练习记录、评分、复习状态和薄弱分类支撑建议。
3. 主动回忆优先：先答题再看答案，避免把产品变成答案浏览器。
4. 表现力服务理解：GSAP 和视觉效果只用于强化流程、状态和空间感。
5. 通用领域模型：Java 是第一版内容，不是系统边界。

## Accessibility

默认支持移动端稳定布局、键盘可访问焦点、清晰对比度和 `prefers-reduced-motion`。练习输入、错误提示、登录注册、复习列表等高频路径需要优先保证可读性和低干扰。
