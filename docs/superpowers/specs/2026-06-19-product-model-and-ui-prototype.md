# 面试雷达产品建模与纯前端原型规格

状态：待审查  
日期：2026-06-19

> 2026-06-20 修订说明：旧 UI 草稿已废弃，本文件只保留产品建模和前端验证范围，不代表已确认视觉方案或已确认技术架构。技术路线见 `2026-06-20-technical-architecture.md`，UI 风格见 `2026-06-20-ui-style-direction.md`。

## 1. 当前决策

面试雷达先做成通用技术面试学习平台，不再把第一版写死成 Java 后端项目。第一阶段只落实产品建模和纯前端 UI 原型，用假数据验证产品形态；后端、真实 AI 调用、持久化、RAG 和 LiteLLM 接入都放到原型审查通过之后。

AI 接入采用 BYOK 方向：用户提供自己的 API Key、Base URL 和模型名，后端阶段推荐用 LiteLLM Proxy 统一适配不同供应商。new-api 只保留为 OpenAI-compatible 网关的一种可选参考，不改源码、不共享数据库、不做单点登录。

## 2. 产品目标

帮助准备技术面试的用户建立自己的面试题库，通过主动回答、AI 反馈和复习计划掌握面试知识与表达技巧。

产品成功不是用户能和 AI 聊天，也不是用户一题一题维护题库，而是用户每天打开后知道：

- 今天该练哪些题。
- 哪些方向最薄弱。
- 自己回答漏了什么。
- 下一次什么时候回来复习。
- 自己的目标岗位或上传资料已经沉淀出了哪些可练题目。

## 3. 不做什么

这个产品长期也不做平台内支付、充值、套餐。AI 调用成本由用户自己的模型供应商或外部网关承担，面试雷达只做学习产品和调用入口。

第一阶段不做这些能力：

- 不做真实后端。
- 登录、注册、找回密码页面要有；纯 UI 阶段可以假登录，接后端时使用 Next.js + PostgreSQL 自己承载用户和会话。
- 不做真实 AI 请求。
- 不接入真实 LiteLLM 或模型供应商。
- 不做支付、充值、套餐，以后也不做。
- 不做多人协作和公开题库社区。
- 不做实时面试辅助或面试中提示。
- 不做 new-api 源码改造、统一用户表或共享数据库。
- 不要求用户一题一题手动创建完整题库。

这些暂时都只在 UI 上保留入口或状态。

## 4. 第一阶段用户路径

```text
选择目标方向
  -> 查看学习工作台
  -> 直接输入想面试的岗位，或上传复习资料、简历、JD、项目笔记
  -> AI 根据岗位目标或资料生成候选题目
  -> 用户批量确认、筛选、修正题目
  -> 选择题目开始练习
  -> 先主动回答，再查看参考答案
  -> 查看 AI 反馈、追问、遗漏点
  -> 加入错题和复习计划
  -> 回到 Dashboard 看今日复习
```

第一阶段可以用固定用户和假数据跑完整流程。

## 5. 核心产品对象

| 对象 | 含义 | 第一阶段字段 | 原型行为 |
| --- | --- | --- | --- |
| User | 当前使用者 | id, name, email | 纯 UI 阶段假登录，后端阶段来自 PostgreSQL 用户表 |
| InterviewTarget | 面试目标 | direction, role, level, deadline | 用户可在 UI 里切换方向 |
| KnowledgeArea | 知识点分组 | id, name, parentId, direction, mastery | 展示知识树和薄弱点 |
| TargetBrief | 岗位目标描述 | role, companyType, techStack, level, focus | 用户直接描述想面试的岗位 |
| SourceDocument | 用户资料 | id, type, title, status, extractedAreas | 上传复习资料、简历、JD、项目笔记 |
| GeneratedQuestionSet | AI 生成题组 | id, sourceType, sourceId, targetId, count, status | 展示批量生成和确认流程 |
| Question | 面试题 | id, title, type, difficulty, tags, source, status | 列表、筛选、详情、少量编辑 |
| AnswerVersion | 答案版本 | id, questionId, style, content, source | 展示标准答案、口语答案、深入答案 |
| PracticeAttempt | 一次练习 | id, questionId, answerText, duration, selfRating | 练习页提交后生成假反馈 |
| AIReview | AI 反馈 | score, missingPoints, suggestions, followups | 展示评分、遗漏点、追问 |
| ReviewState | 复习状态 | questionId, mastery, dueAt, mistakeCount | Dashboard 和错题页使用 |
| AIProviderConfig | 用户 AI 接入配置 | provider, baseUrl, model, status | 展示 BYOK 配置原型 |

## 6. 题目来源模型

题目来源有两个主入口：

- 岗位目标生成：用户直接说想面试的岗位、方向、级别、技术栈或公司类型，AI 批量生成候选题和答案草稿。
- 资料增强生成：用户上传复习资料、简历、JD、项目笔记，AI 基于资料生成更个性化的候选题和追问。
- 批量导入：用户粘贴 Markdown、CSV、JSON 或文本题库，AI 帮忙清洗成结构化题目。

用户可以编辑 AI 生成结果，但编辑是修正动作，不是主创建方式。第一阶段只做 UI 表现。AI 生成和资料生成都返回假数据，但界面要让用户看懂未来工作流。

## 7. 练习模型

练习页必须围绕主动回忆设计：

- 默认隐藏答案。
- 用户先输入自己的回答。
- 用户可以标记“不熟、一般、掌握”。
- 提交后展示 AI 反馈。
- AI 反馈包括评分、遗漏点、表达建议、追问和推荐复习时间。

答案浏览可以存在，但不能变成主流程。

## 8. 页面范围

第一阶段纯前端原型的顶级导航只包含这些页面：

| 页面 | 目的 | 关键内容 |
| --- | --- | --- |
| Dashboard | 每天打开后的学习工作台 | 今日待复习、薄弱方向、最近练习、AI 接入轻状态 |
| AI 生成 | 从岗位或资料构建题库 | 岗位描述、资料选择、难度、数量、候选题组、批量确认 |
| 题库 | 管理个人题库 | 题目列表、筛选、标签、来源、掌握度 |
| 练习 | 主动回答题目 | 题干、输入区、答案开关、提交、反馈 |
| 资料 | 管理复习资料 | 复习资料、简历、JD、项目笔记、关联题目 |
| 设置 | 用户配置 | AI Provider 配置入口 |

登录、注册、找回密码不放进工作台顶级导航，但第一版必须有独立页面。

目标方向、AI 补题、反馈报告不做独立顶级页面：

- 目标方向放在 Dashboard 顶部和 AI 生成页表单里。
- AI 补题放在 AI 生成页的“补薄弱点”模式里。
- 反馈报告放在练习提交后的结果状态里。

移动端和桌面端共用一套 Web/PWA 体验。

## 8.1 第一版范围锁定

为避免反复回退，第一版默认锁定：

- 做登录、注册、找回密码页面；纯 UI 阶段用假登录进入 Dashboard。
- 不做真实上传解析，资料页只 mock 文件选择、上传状态和解析结果。
- 不做语音回答，练习只做文本回答。
- 不做 PWA 离线、Service Worker、安装提示；先做响应式 Web。
- 不做 localStorage，刷新后回到初始 mock 状态。
- 不做完整批量导入编辑器，只在资料或 AI 生成页保留“粘贴题库文本”的轻入口。
- 第一版方向模板包含：前端、Java 后端、全栈、算法、系统设计、项目经历、行为面试。
- 第一版题目类型包含：技术概念题、项目追问题、系统设计题、算法思路题、行为/STAR 题。
- AI 反馈维度固定为：完整度、深度、结构化表达、遗漏点、追问风险、下次复习建议。

## 9. AI 接入边界

第一阶段只保留 BYOK 配置原型：

```text
设置 / AI 接入
  - 供应商格式
  - Base URL
  - API Key
  - 默认模型
  - 未配置状态
  - 简短说明：用户使用自己的模型供应商额度，面试雷达不做平台额度体系
```

第一阶段不真实保存 Base URL、API Key、模型偏好和用量摘要，也不固定 `http://localhost:3000`。如果 UI 需要按钮，默认显示未配置或禁用态。

后续真实接入时再单独确认调用链，候选方案是：

```text
浏览器
  -> 面试雷达后端
  -> 读取当前用户加密保存的 AI Provider 配置
  -> 调用 LiteLLM Proxy
  -> LiteLLM 路由到 OpenAI / Claude / Gemini / DeepSeek / 自建 OpenAI-compatible 网关
```

生产环境不让浏览器直接持有可调用的 API Key。API Key 由后端加密保存，等后续接入规格再定。

如果以后仍要接 new-api，优先把它当 OpenAI-compatible 网关或独立工具，不改 new-api 用户表。

## 10. 参考产品与取舍

### OpenExam

本地参考项目：`D:\mianshi\.reference\OpenExam`

可借鉴：

- 资料/题库导入后再练习，而不是要求用户逐题维护。
- AI 出卷、专项练习、错题本、练习历史、统计分析组成闭环。
- 答题页保持专注，结果页再展示解析和知识点。
- 设置页只承载外部 AI 能力、数据备份、导入等低频配置。

不直接照搬：

- OpenExam 偏考试/试卷，本项目偏面试表达和知识掌握，不把“试卷中心”做成主入口。
- 成就、等级、学习日历可以以后再加，第一阶段不做游戏化。
- 桌面本地 SQLite 不是当前方向，本项目仍是 Web/PWA。

### 线上参考

- Yoodli：可借鉴“选择场景 -> 开始练习 -> AI 追问 -> 查看结果”的闭环，以及内容、表达、进步趋势反馈；先不做多人角色扮演和企业培训管理。参考：https://yoodli.ai/
- Final Round AI：可借鉴“选择目标岗位 / 上传简历 -> 个性化模拟面试 -> 反馈报告”；明确不做实时隐形面试辅助。参考：https://www.finalroundai.com/
- Grow with Google 面试准备指南：可借鉴问题分类、STAR 方法、角色研究和反复练习；产品里可以把行为题、情景题、技术题分开建模。参考：https://grow.google/grow-your-career/articles/interview-tips/

## 11. 前端验证要求

前端验证目标是判断产品是不是想要的，不追求后端真实。旧 UI 已废弃，下一版需要在技术架构和 UI 风格确认后重新实现。

要求：

- 所有页面可从主导航进入。
- 关键按钮有反馈状态，即使是假数据。
- 桌面端能高效扫视信息。
- 移动端能完成练习和查看反馈。
- 页面文案以学习动作命名，不写成技术架构展示。
- 设置页只展示 AI Provider 配置原型，不在前端真实保存 API Key。

第一版 UI 可以用 mock 数据演示。真实持久化以后端数据库为准，不把 localStorage 当正式方案。

## 12. 验收标准

文档审查通过后，纯前端 UI 原型需要满足：

- 用户能从 Dashboard 进入一次完整练习流程。
- 用户能看到 Dashboard、AI 生成、题库、练习、资料、设置六个核心入口。
- 用户能理解题库主要来自岗位目标、资料上传和 AI 批量生成，不需要逐题创建。
- 用户能理解 LiteLLM 是后端统一适配层，自己只需要配置供应商、Base URL、API Key 和模型。
- 用户能在桌面端和移动端判断整体信息架构是否顺手。
- 原型不依赖数据库、后端服务或真实 AI key。
